## 1. Overview

### What this feature does
A prospective member fills out a registration form on `become-a-member.html`. On submission:

1. The form data is **validated** on the frontend (UX) and again on the **backend** (security).
2. The data is **stored in MySQL** as a pending membership request.
3. **Two emails are sent** via PHPMailer + IONOS SMTP:
   - An internal notification email to Bling staff.
   - A confirmation email to the applicant.
4. The user sees a **success message** without any page reload.

### What this feature does NOT do
- It does not approve or reject memberships automatically.
- It does not create user accounts or logins.
- It does not process payments.

### Visual & UX requirements
The page must follow the exact same design system as the rest of the site:
- Font: **Montserrat** (already loaded via Google Fonts)
- Colors: `--blue: #030081`, `--green: #d1d70d`, `--white: #ffffff`
- Custom cursor (diamond shape), scroll reveal animations, nav + footer identical to other pages
- Form submission via `fetch()` — **no page reload**
- Loading state on the submit button during processing
- Success and error states displayed inline

### Critical constraints
- **No JavaScript frameworks** — vanilla JS only, matching the existing codebase.
- **No page reloads on form submission** — use `fetch()` API exclusively.
- **PDO with prepared statements** — all database queries must use parameterized statements.
- **PHPMailer via Composer** — never use PHP's native `mail()` function.
- **`display_errors` must be `Off` in production** — errors must be logged, never printed.

## 4. Database Schema

### Table: `membership_requests`

```sql
CREATE TABLE membership_requests (
  id                  INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,

  -- Company Information
  company_name        VARCHAR(255)    NOT NULL,
  company_country     VARCHAR(100)    NOT NULL,
  company_city        VARCHAR(100)    NOT NULL,
  company_address     VARCHAR(500)    DEFAULT NULL,
  company_website     VARCHAR(255)    DEFAULT NULL,
  years_in_business   TINYINT UNSIGNED DEFAULT NULL,

  -- Primary Contact
  contact_first_name  VARCHAR(100)    NOT NULL,
  contact_last_name   VARCHAR(100)    NOT NULL,
  contact_email       VARCHAR(255)    NOT NULL,
  contact_phone       VARCHAR(50)     NOT NULL,
  contact_position    VARCHAR(100)    DEFAULT NULL,

  -- Services offered (stored as JSON array)
  services_offered    JSON            DEFAULT NULL,

  -- Reference (optional — another member who referred them)
  referred_by         VARCHAR(255)    DEFAULT NULL,

  -- Message / Additional notes
  message             TEXT            DEFAULT NULL,

  -- Privacy & Legal
  privacy_accepted    TINYINT(1)      NOT NULL DEFAULT 0,

  -- Metadata
  status              ENUM('pending','reviewing','approved','rejected')
                                      NOT NULL DEFAULT 'pending',
  ip_address          VARCHAR(45)     NOT NULL,        -- supports IPv6
  user_agent          VARCHAR(500)    DEFAULT NULL,
  recaptcha_score     DECIMAL(3,2)    DEFAULT NULL,
  submitted_at        DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at          DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP
                                      ON UPDATE CURRENT_TIMESTAMP,

  -- Indexes
  INDEX idx_status         (status),
  INDEX idx_submitted_at   (submitted_at),
  INDEX idx_contact_email  (contact_email)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### Table: `rate_limit_log`

```sql
CREATE TABLE rate_limit_log (
  id           INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  ip_address   VARCHAR(45)  NOT NULL,
  endpoint     VARCHAR(100) NOT NULL DEFAULT 'membership',
  attempted_at DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_ip_endpoint (ip_address, endpoint),
  INDEX idx_attempted_at (attempted_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

> **Rate limit rule:** Max **3 submissions per IP per 10 minutes** per endpoint.
> Records older than 24 hours can be purged periodically.

### 5.3 `js/become-a-member.js`

**Purpose:** Client-side form validation and async form submission.

**Dependencies:** `base.js` must be loaded before this file.

**Functions to implement:**

#### `initForm()`
- Called on DOMContentLoaded
- Attaches event listeners to the form submit event
- Calls `initCursorHover()` from `base.js` with form-specific selectors

#### `validateField(field)`
- Validates a single field
- Returns `{ valid: boolean, message: string }`
- Called on `blur` event for each field (real-time feedback)
- Rules:
  - Required fields: must not be empty after `.trim()`
  - `contact_email`: must match `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`
  - `contact_phone`: must contain only digits, spaces, `+`, `-`, `()`, min 7 chars
  - `company_website`: if provided, must start with `http://` or `https://`
  - `privacy_accepted`: must be checked

#### `validateForm()`
- Calls `validateField()` on all fields
- Returns `true` only if all fields pass
- Displays error messages next to invalid fields

#### `getRecaptchaToken()`
- Calls `grecaptcha.execute(SITE_KEY, {action: 'membership_submit'})`
- Returns a Promise that resolves to the token string

#### `serializeForm(form)`
- Collects all form field values including the services checkbox array
- Returns a plain object (not FormData) to be JSON-stringified
- Excludes the honeypot field from serialization

#### `handleHoneypot(data)`
- Checks if the honeypot field has a value
- If yes: simulates success (show success message, do not send request) → returns `true`
- If no: returns `false` (proceed normally)

#### `submitForm(data, token)`
- Sends `POST` to `api/submit-membership.php`
- Body: `JSON.stringify({ ...data, recaptcha_token: token })`
- Headers: `{ 'Content-Type': 'application/json', 'X-Requested-With': 'XMLHttpRequest' }`
- Returns the parsed JSON response

#### `showSuccess(email)`
- Hides the form
- Shows the success message with the applicant's email interpolated

#### `showError(message)`
- Shows the error message below the submit button
- Re-enables the submit button

#### `handleSubmit(event)`
- Prevents default form submission
- Checks honeypot → if triggered, silently simulate success and return
- Calls `validateForm()` → if invalid, stop and return
- Sets button to loading state
- Gets reCAPTCHA token
- Calls `submitForm()` with form data + token
- On success (status === 'success'): calls `showSuccess()`
- On error or network failure: calls `showError()`

### 5.6 `api/submit-membership.php`

**Purpose:** The single PHP endpoint that receives, validates, stores, and emails the membership request.

**Request contract:**
- Method: `POST`
- Content-Type: `application/json`
- Body: JSON object with all form fields + `recaptcha_token`

**Response contract:**
- Content-Type: `application/json`
- Success: `{ "status": "success", "message": "Application received." }` — HTTP 200
- Validation error: `{ "status": "error", "message": "...", "errors": {} }` — HTTP 422
- Rate limited: `{ "status": "error", "message": "Too many requests." }` — HTTP 429
- Server error: `{ "status": "error", "message": "Server error." }` — HTTP 500

**Processing flow (implement in this exact order):**

```
1. BOOTSTRAP
   ├── define('BLING_APP', true)
   ├── require config.php
   ├── Set response header: Content-Type: application/json
   ├── Set CORS header (same origin only): only allow requests from APP_URL
   └── Verify request method is POST → 405 if not

2. PARSE INPUT
   ├── Read php://input and json_decode
   └── If JSON is invalid → 400 Bad Request

3. SAME-ORIGIN CHECK
   └── Verify X-Requested-With header === 'XMLHttpRequest' → 403 if missing

4. RATE LIMITING
   ├── Get client IP (use $_SERVER['HTTP_X_FORWARDED_FOR'] if behind proxy, else REMOTE_ADDR)
   ├── Query rate_limit_log: count rows WHERE ip = ? AND endpoint = 'membership'
   │   AND attempted_at > NOW() - INTERVAL RATE_LIMIT_WINDOW MINUTES
   ├── If count >= RATE_LIMIT_MAX → 429 Too Many Requests
   └── INSERT new log entry for this IP

5. RECAPTCHA VERIFICATION
   ├── POST to https://www.google.com/recaptcha/api/siteverify
   │   with secret=RECAPTCHA_SECRET and response=recaptcha_token
   ├── If success !== true → reject
   └── If score < RECAPTCHA_MIN_SCORE → reject silently (return fake success)

6. HONEYPOT CHECK
   └── If 'website_url' field is not empty → reject silently (return fake success)

7. INPUT VALIDATION (backend — independent of frontend)
   Validate each field. Collect all errors before returning.
   Rules:
   ├── company_name:       required, max 255 chars
   ├── company_country:    required, max 100 chars
   ├── company_city:       required, max 100 chars
   ├── company_address:    optional, max 500 chars
   ├── company_website:    optional, must be valid URL if provided
   ├── years_in_business:  optional, integer 0-150
   ├── contact_first_name: required, max 100 chars
   ├── contact_last_name:  required, max 100 chars
   ├── contact_email:      required, must pass filter_var FILTER_VALIDATE_EMAIL
   ├── contact_phone:      required, min 7 chars, only digits/spaces/+/-()/
   ├── contact_position:   optional, max 100 chars
   ├── services_offered:   optional, must be array if provided, each item max 50 chars
   ├── referred_by:        optional, max 255 chars
   ├── message:            optional, max 2000 chars
   └── privacy_accepted:   must be true (boolean) or '1' or 1
   If any errors → return 422 with errors object

8. SANITIZE
   └── Run htmlspecialchars() on all string inputs before storing

9. DATABASE INSERT
   ├── Open PDO connection (DSN from config)
   ├── PDO::ATTR_ERRMODE → PDO::ERRMODE_EXCEPTION
   ├── PDO::ATTR_EMULATE_PREPARES → false
   ├── Prepare INSERT statement for membership_requests
   ├── Bind all parameters (never concatenate)
   ├── Execute
   ├── Get lastInsertId() for reference
   └── On PDOException → log error, return 500

10. SEND INTERNAL NOTIFICATION EMAIL
    ├── Build HTML email with all submitted data (see Email Templates section)
    ├── Use PHPMailer with SMTP credentials from config
    ├── Send to each address in NOTIFY_EMAILS
    ├── Reply-To: contact_email (so staff can reply directly to the applicant)
    └── On failure → log error but do NOT stop execution (still send confirmation)

11. SEND CONFIRMATION EMAIL TO APPLICANT
    ├── Build HTML email confirming receipt (see Email Templates section)
    ├── Send to contact_email
    └── On failure → log error

12. RETURN SUCCESS
    └── Return 200 { "status": "success", "message": "Application received." }
```

## 7. Security Requirements

These are non-negotiable. Every item must be implemented.

### Input Security
| Requirement | Implementation |
|---|---|
| SQL Injection prevention | PDO prepared statements on every query — no exceptions |
| XSS prevention | `htmlspecialchars($value, ENT_QUOTES, 'UTF-8')` on all output |
| CSRF mitigation | `X-Requested-With: XMLHttpRequest` header check + same-origin policy |
| Input length limits | Enforced in both frontend (maxlength attr) and backend (PHP strlen check) |

### Anti-Bot
| Requirement | Implementation |
|---|---|
| reCAPTCHA v3 | Token verified server-side, score ≥ 0.5 required |
| Honeypot | Invisible field; any value = silent discard |
| Rate limiting | 3 requests / IP / 10 min, tracked in `rate_limit_log` table |

### File & Server Security
| Requirement | Implementation |
|---|---|
| No direct access to `/api/` | `.htaccess` with `Deny from all` |
| No direct access to `/vendor/` | `.htaccess` with `Deny from all` |
| Config file protection | `BLING_APP` constant check at top of config.php |
| No error display in production | `display_errors = Off`, `log_errors = On` |
| HTTPS only | Root `.htaccess` redirects HTTP → HTTPS |

### Email Security
| Requirement | Implementation |
|---|---|
| SPF record | Configured in IONOS DNS panel |
| DKIM record | Configured in IONOS DNS panel |
| No `mail()` function | PHPMailer via SMTP exclusively |
